# 设计检查功能使用指南

## 📋 功能概述

设计检查功能是一个全面的架构设计质量评估工具，可自动检测系统设计中的问题并提供改进建议。

## 🎯 核心检查维度

### 1. 架构规则验证 (Architecture Rule)
**权重：25%**

检查系统架构是否符合行业标准和最佳实践：

- ✅ **必须有前端应用** - 系统应该至少包含一个前端节点
- ✅ **必须有数据库** - 系统应该至少包含一个数据库节点
- ✅ **API 需要 Service 支持** - API 节点应该连接到 Service 节点
- ✅ **Service 需要数据层** - Service 节点应该连接到 Repository 或 Database
- ✅ **避免循环依赖** - 检测违反分层架构的反向连接

### 2. 数据完整性检查 (Data Integrity)
**权重：20%**

检查节点配置的完整性和准确性：

- ✅ **节点名称完整性** - 所有节点应该有意义的名称（不能是"示例"）
- ✅ **关键配置完整性** - 检查各类节点的关键配置参数
  - Frontend: framework, port
  - API: route, method
  - Database: type, host
- ✅ **描述信息完整性** - 关键节点（API、Service、Agent）应该有详细描述

### 3. 连接匹配验证 (Connection Matching)
**权重：20%**

检查节点间连接关系的合理性：

- ✅ **孤立节点检测** - 发现未连接的节点（Frontend 和 Database 除外）
- ✅ **重复连接检测** - 发现两个节点间的重复连接
- ✅ **连接方向正确性** - 确保数据流向符合分层架构

### 4. 性能检查 (Performance)
**权重：15%**

评估系统性能相关的潜在问题：

- ⚠️ **单点故障风险** - 检测只有一个 Service 节点的情况
- ⚠️ **数据库连接过载** - 检测单个数据库被过多 Repository 连接

### 5. 安全检查 (Security)
**权重：15%**

识别安全隐患：

- 🔒 **API 认证检查** - 检测未启用认证的 API 节点
- 🔒 **数据库暴露风险** - 检测 Frontend 直接连接 Database 的严重问题

### 6. 可扩展性检查 (Scalability)
**权重：5%**

评估系统的可扩展性设计：

- 📈 **Repository 模式使用** - 鼓励使用 Repository 层封装数据访问
- 📈 **缓存设计** - 未来将检查是否使用缓存层

## 📊 评分体系

### 总体评分
- **优秀 (90-100 分)** ⭐⭐⭐⭐⭐ - 设计质量卓越，符合所有最佳实践
- **良好 (75-89 分)** ⭐⭐⭐⭐ - 设计合理，有少量可改进之处
- **一般 (60-74 分)** ⭐⭐⭐ - 设计基本合格，存在需要改进的问题
- **需改进 (<60 分)** ⭐⭐ - 设计存在严重问题，需要重新审视

### 问题严重程度
- 🔴 **严重 (Critical)** - 必须立即修复的关键问题
- 🟡 **警告 (Warning)** - 建议修复的潜在问题
- 🔵 **建议 (Info)** - 优化建议，可选择性采纳

## 🚀 使用方法

### 基本步骤

1. **打开设计检查**
   - 点击顶部工具栏的"🔍 设计检查"按钮
   - 系统自动开始全面检查

2. **查看检查报告**
   - 右侧面板显示详细报告
   - 查看总体评分和等级
   - 浏览各维度得分

3. **分析问题**
   - 查看"严重问题"和"警告"列表
   - 了解每个问题的详细描述
   - 查看受影响的节点

4. **改进设计**
   - 根据建议进行修复
   - 重新运行检查验证改进效果

### 示例场景

#### 场景 1: 新建系统架构检查
```
1. 创建基础架构（Frontend → API → Service → Database）
2. 点击"设计检查"
3. 查看评分（预计 80-90 分）
4. 根据建议完善配置
```

#### 场景 2: 发现安全隐患
```
问题：Frontend 直接连接 Database
检查发现：🔴 数据库直接暴露给前端（严重问题）
修复：添加 API 层和 Service 层
重新检查：评分提升
```

#### 场景 3: 优化数据完整性
```
问题：多个节点使用"示例"名称
检查发现：🟡 节点名称缺失
修复：为所有节点设置有意义的名称
重新检查：数据完整性维度得分提升
```

## 📝 常见问题

### Q1: 为什么我的评分很低？
**A:** 检查报告中的"问题详情"部分会列出所有问题，逐一修复即可提升评分。

### Q2: 某些警告需要修复吗？
**A:** 
- 🔴 严重问题：必须修复
- 🟡 警告：建议修复
- 🔵 建议：可选择性采纳

### Q3: 如何提升安全评分？
**A:** 
1. 为 API 启用认证（requiresAuth: true）
2. 确保 Frontend 不直接连接 Database
3. 考虑添加认证中间件

### Q4: 检查功能会影响现有设计吗？
**A:** 不会。检查功能只读，不会修改任何节点或连接。
