# 📦 模块管理器使用指南

## 📖 概述

模块管理器是 Meta-Arch 架构设计器的核心功能之一，用于管理和组织架构中的模块及其依赖关系。通过模块管理，您可以：

- ✅ 创建和管理架构模块
- ✅ 定义模块类型和层级
- ✅ 配置模块依赖关系
- ✅ 验证模块设计规范
- ✅ 生成依赖分析报告

---

## 🎯 核心概念

### 1. 模块类型 (ModuleType)

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **core** | 核心模块 | 系统的核心业务逻辑 |
| **feature** | 功能模块 | 特定功能模块 |
| **shared** | 共享模块 | 跨模块共享的工具和组件 |
| **infrastructure** | 基础设施模块 | 数据库、消息队列等基础设施 |
| **api** | API 模块 | 对外提供的 API 接口 |
| **ui** | UI 模块 | 用户界面组件 |
| **domain** | 领域模块 | 领域驱动设计的领域模块 |
| **application** | 应用模块 | 应用层服务模块 |

### 2. 架构层级 (ModuleLayer)

| 层级 | 说明 | 依赖规则 |
|------|------|----------|
| **presentation** | 展示层 | 可依赖应用层 |
| **application** | 应用层 | 可依赖领域层 |
| **domain** | 领域层 | 可依赖基础设施层 |
| **infrastructure** | 基础设施层 | 不依赖其他层 |

**依赖规则**: 上层可以依赖下层，下层不能依赖上层

## 🚀 使用步骤

### 步骤 1: 打开模块管理器

1. 启动应用后，点击顶部工具栏的 **"📦 模块管理"** 按钮
2. 模块管理面板将在右侧打开

### 步骤 2: 创建第一个模块

1. 点击 **➕ 创建新模块** 按钮
2. 填写模块信息：

```
模块名称：UserModule
模块类型：domain
架构层级：domain
描述：用户管理领域模块，负责用户相关的业务逻辑
```

3. 点击 **✅ 创建** 按钮
4. 系统会自动验证模块配置

### 步骤 3: 创建依赖模块

继续创建其他模块，例如：

```
模块名称：UserRepository
模块类型：infrastructure
架构层级：infrastructure
描述：用户数据仓库模块

模块名称：UserService
模块类型：application
架构层级：application
描述：用户应用服务模块

模块名称：UserComponent
模块类型：ui
架构层级：presentation
描述：用户界面组件模块
```

### 步骤 4: 查看模块详情

1. 点击模块列表中的任意模块
2. 在底部查看模块详情：
   - 模块 ID
   - 版本号
   - 依赖数量
   - 导出数量

### 步骤 5: 生成依赖报告

1. 选中一个模块
2. 点击 **📄 查看依赖报告**
3. 在控制台查看详细的依赖关系分析

## 📋 最佳实践

### ✅ 推荐做法

1. **分层清晰**: 确保模块归属正确的架构层级
2. **单一职责**: 每个模块只负责一个明确的业务领域
3. **依赖向下**: 遵循依赖
   ```
   presentation → application → domain → infrastructure
   ```
4. **命名规范**: 使用一致的命名约定
   - 领域模块：`XxxDomain` 或 `XxxModule`
   - 服务模块：`XxxService`
   - 仓库模块：`XxxRepository`
   - 组件模块：`XxxComponent`

### ❌ 避免做法

1. **循环依赖**: 模块 A 依赖模块 B，模块 B 又依赖模块 A
2. **跨层依赖**: 展示层直接依赖基础设施层
3. **模块过大**: 一个模块包含过多职责
4. **命名混乱**: 同一类型的模块使用不同的命名风格

---

## 🔍 模块验证规则

系统会自动验证以下规则：

### 1. 命名规则
- ✅ 必须以字母开头
- ✅ 只能包含字母、数字和下划线
- ✅ 长度在 2-50 个字符之间
- ✅ 建议使用 PascalCase（大驼峰命名）

### 2. 层级规则
- ✅ 展示层 → 应用层 ✅
- ✅ 应用层 → 领域层 ✅
- ✅ 领域层 → 基础设施层 ✅
- ❌ 基础设施层 → 领域层 ❌ (违规)
- ❌ 领域层 → 应用层 ❌ (违规)

### 3. 依赖规则
- ✅ 模块不能依赖自身
- ✅ 不能形成循环依赖
- ✅ 依赖的模块必须存在

## 🛠️ 常见问题

### Q1: 为什么要使用模块管理？

**A**: 模块管理帮助您：
- 清晰地组织代码结构
- 避免循环依赖和架构混乱
- 提高代码可维护性和可测试性
- 便于团队协作和代码审查

### Q2: 模块和节点有什么区别？

**A**: 
- **节点**: 架构设计图中的视觉元素（Frontend、API、Service 等）
- **模块**: 代码组织的逻辑单元，用于实际的项目结构

两者可以相互映射，例如一个 Service 节点可能对应多个模块。

### Q3: 如何导出模块配置？

**A**: 
1. 模块管理功能完善后，将支持导出为 JSON 文件
2. 可以与其他团队成员分享模块配置
3. 可以导入到其他项目中复用

### Q4: 模块验证失败怎么办？

**A**: 
1. 查看验证错误信息
2. 根据错误提示修改模块配置
3. 常见错误：命名不规范、层级错误、依赖不存在
